# Настройка постоянного SSH-агента через systemd

В этом руководстве описана настройка `ssh-agent` как постоянного пользовательского сервиса systemd. Это позволяет использовать SSH-агент независимо от терминальных сессий и гарантирует, что контейнеры Docker, запущенные через `docker compose`, смогут использовать агент для аутентификации на удалённых узлах.

---

## Предварительные условия

- Операционная система Linux с systemd (проверено на Ubuntu)
- У вас есть приватные SSH-ключи в `~/.ssh/` (например, `id_rsa`, `id_ed25519`)
- Установлен Docker и Docker Compose

---

## 1. Создание systemd-сервиса для ssh-agent

Создайте файл сервиса в пользовательском каталоге systemd:

```bash
mkdir -p ~/.config/systemd/user
nano ~/.config/systemd/user/ssh-agent.service
```

Вставьте следующее содержимое:

```ini
[Unit]
Description=SSH key agent

[Service]
Type=simple
Environment=SSH_AUTH_SOCK=%t/ssh-agent.socket
ExecStart=/usr/bin/ssh-agent -D -a $SSH_AUTH_SOCK

[Install]
WantedBy=default.target
```

**Пояснение:**

- `%t` — системная переменная systemd, указывающая на временную директорию пользователя (`/run/user/<UID>`).
- Опция `-D` запускает агент в режиме демона, `-a` задаёт путь к сокету.

---

## 2. Настройка переменной окружения

Чтобы переменная `SSH_AUTH_SOCK` автоматически устанавливалась при входе в систему, создайте файл конфигурации окружения systemd:

```bash
mkdir -p ~/.config/environment.d
nano ~/.config/environment.d/ssh_auth_socket.conf
```

Добавьте строку:

```bash
SSH_AUTH_SOCK="${XDG_RUNTIME_DIR}/ssh-agent.socket"
```

Эта настройка гарантирует, что `SSH_AUTH_SOCK` будет доступна во всех будущих сессиях пользователя.

---

## 3. Запуск и включение сервиса

Выполните команды для перезагрузки конфигурации systemd, запуска сервиса и его включения при старте сессии:

```bash
systemctl --user daemon-reload
systemctl --user enable --now ssh-agent
```

Проверьте статус сервиса:

```bash
systemctl --user status ssh-agent
```

Вы должны увидеть `active (running)` и путь к сокету.

---

## 4. Добавление ключей в агент

Агент работает, но пока не содержит ни одного ключа. Добавьте ваши приватные ключи:

```bash
ssh-add ~/.ssh/id_rsa
ssh-add ~/.ssh/id_ed25519   # если используется
```

Если ключи защищены парольной фразой, система запросит её ввод.

Проверьте список ключей:

```bash
ssh-add -l
```

Вывод должен содержать отпечатки добавленных ключей.

### (Опционально) Автоматическое добавление ключей при первом использовании

Чтобы не выполнять `ssh-add` вручную при каждой новой сессии, добавьте в ваш `~/.ssh/config` следующую строку:

```ssh-config
AddKeysToAgent yes
```

При первом SSH-подключении ключ будет автоматически добавлен в агент (потребуется ввод пароля, если он есть).

---

## 5. Проверка работоспособности

Проверьте, что агент доступен и ключи работают:

```bash
ssh -T git@github.com   # пример для GitHub
```

или для любого другого хоста:

```bash
ssh -o ForwardAgent=yes user@host "echo OK"
```

Если команда выполняется успешно, агент настроен корректно.

---

## 6. Интеграция с Docker Compose

В вашем `docker-compose.yml` должно быть монтирование сокета агента:

```yaml
services:
  bloodhound_gang:
    ...
    environment:
      - HOST_UID=${UID:-1000}
      - HOST_GID=${GID:-1000}
      - SSH_AUTH_SOCK=/ssh-auth.sock   # необязательно, если не используется внутри контейнера
    volumes:
      - ${SSH_AUTH_SOCK}:/ssh-auth.sock:ro
    ...
```

Убедитесь, что переменная `SSH_AUTH_SOCK` подставляется из окружения (она уже будет установлена после настройки systemd и `.bashrc`).

Внутри контейнера команды `ssh` должны использовать опцию `-o ForwardAgent=yes` (или быть настроены с `ForwardAgent yes` в `~/.ssh/config` внутри контейнера). В вашем Python-коде это должно быть учтено (в `ssh_cmd` уже присутствует `-o ForwardAgent=yes`).

---

## 7. Обновление скрипта запуска (run.sh)

Поскольку агент теперь работает как постоянный сервис, ваш `run.sh` больше не должен запускать `ssh-agent` вручную. Удалите строки с `eval $(ssh-agent -s)` и `ssh-add`. Минимальный скрипт:

```bash
#!/bin/bash
export HOST_UID=$(id -u)
export HOST_GID=$(id -g)
export SSH_USER=$(whoami)
docker compose up -d --pull always
```

---

## Устранение неполадок

### 1. Переменная `SSH_AUTH_SOCK` не появляется в терминале

- **Причина:** Окружение systemd не импортировано или оболочка не является логин-шеллом.
- **Решение:**
  - Добавьте экспорт в `~/.bashrc` (или `~/.zshrc`):
    ```bash
    export SSH_AUTH_SOCK="/run/user/$(id -u)/ssh-agent.socket"
    ```
  - Затем выполните `source ~/.bashrc` или откройте новый терминал.

### 2. Команда `ssh-add -l` выводит `Could not open a connection to your authentication agent`

- **Причина:** Переменная `SSH_AUTH_SOCK` не установлена или указывает на несуществующий сокет.
- **Решение:**
  - Проверьте `echo $SSH_AUTH_SOCK`; если пусто, экспортируйте вручную (см. пункт 1).
  - Убедитесь, что сокет существует: `ls -la /run/user/$(id -u)/ssh-agent.socket`.

### 3. Агент запущен, но ключи не добавляются

- **Причина:** Ключи не добавлены явно, либо `AddKeysToAgent` не настроен.
- **Решение:** Выполните `ssh-add` и проверьте `ssh-add -l`.

### 4. Из контейнера не удаётся подключиться, хотя агент работает на хосте

- **Причина:** Сокет смонтирован, но внутри контейнера пользователь не имеет доступа к сокету.
- **Решение:**
  - Убедитесь, что в compose используется `:ro` (read-only) и что UID внутри контейнера совпадает с UID хоста (вы передаёте `HOST_UID` и используете его для создания пользователя внутри контейнера или через `user:`).
  - Проверьте права на сокет на хосте: `ls -la /run/user/$(id -u)/ssh-agent.socket` — должно быть `srw-------` с владельцем вашего пользователя.

### 5. После перезагрузки системы агент не запускается

- **Причина:** Сервис включён, но systemd пользовательский запускается только при входе в систему.
- **Решение:** Убедитесь, что сервис включён: `systemctl --user enable ssh-agent`. Если не запускается автоматически, можно добавить его в автозагрузку через `~/.config/autostart` или просто запускать вручную после входа (но это менее удобно).

---

## Заключение

После выполнения всех шагов `ssh-agent` будет работать как системный сервис, ключи будут доступны для всех ваших сессий и для контейнеров Docker. Это надёжное решение для production-сред и долгоживущих процессов, таких как Nextflow-пайплайны, запускаемые из вашего приложения.

Если у вас возникнут дополнительные вопросы, обратитесь к системному администратору или проверьте журналы systemd:

```bash
journalctl --user -u ssh-agent -f
```
